# Decision Research Agent v0.1.1

## Supported Surface

v0.1.1 aggregates the repository changes merged after v0.1.0 through the
deterministic Agent evaluation gate. Its supported surface includes:

- the FastAPI backend and canonical run/result APIs from v0.1.0;
- the Python Tool Client and its structured local golden flow;
- the separately built React Agent Research Operations Console;
- deterministic downstream-consumer and Agent-evaluation proof commands;
- repository tests, public documentation, and operator scripts.

The application database remains business authority for ResearchRun,
EvidenceLedger, review, verification, publication, and delivery. The console
and proof reports do not acquire that authority.

## Changes

### Structured Tool Client

- Added a structured Tool Client golden flow that creates a run, waits with a
  bounded client deadline, and retrieves the canonical result through
  `--wait --result`.
- Added bounded errors for connection, polling, review-required,
  invalid-response, and result-delivery failures without exposing raw
  exceptions or private response data.

### Agent Research Operations Console

- Added the React Agent Research Operations Console with deterministic Static
  Demo and an optional loopback-only Live Backend flow for
  `health -> run -> canonical result`.
- Added visual and accessibility QA, frontend test/lint/build CI, and public
  setup, architecture, and demo documentation.
- The console does not accept credentials and does not own review,
  verification, or publication authority.

### Deterministic Contract Evidence

- Added a deterministic downstream consumer fixture and validator proof for
  status, canonical result, Evidence, fallback, and stable failure handling.
- Added eight fixed cases and six evaluators to a credential-free Agent
  evaluation regression gate covering result, trajectory, Evidence, terminal
  state, safety, and fixture-defined efficiency boundaries.

### Scoped Maintenance

- Updated the pinned GitHub `setup-node` action and scoped frontend development
  dependencies for `jsdom`, Vite, and Vitest.
- Kept Node compatibility documentation and the deterministic demo-video route
  aligned with the tested frontend surface.

## Compatibility And Migration

No runtime API, schema, or database migration was added between v0.1.0 and
v0.1.1. Existing canonical run, result, Evidence, review, verification, and
publication contracts remain compatible, and this release makes no breaking
contract claim.

No release-specific data migration is required when upgrading from v0.1.0.
Operators using older pre-v0.1.0 databases must still follow the migration and
legacy-table procedures in the [v0.1.0 release notes](v0.1.0.md).

The console is a separate frontend build. It consumes the canonical API and
does not introduce a second runtime or persistence authority.

## Rollback

Rollback is an operator-controlled source and deployment action. Stop the
frontend and backend processes, restore the previous application revision, and
restart only after the usual health and canonical result checks pass.

Because v0.1.1 adds no database migration, it has no release-specific schema
rollback. If operational data must be restored for an independent reason,
restore the application database, checkpoint database, and output storage from
one consistent backup set as documented for v0.1.0.

## Known Limits

- The console is not a publicly hosted service, does not accept credentials,
  and does not own review, verification, or publication authority.
- Static Demo is deterministic fixture presentation, not a live-provider run.
- Live Backend is loopback-only and consumes the existing canonical service;
  it is not an online multi-user deployment.
- The downstream and Agent evaluation proofs are deterministic fixed-input
  contract evidence, not production SLA, not answer-quality accuracy, not real
  users, and not provider measurements.
- Estimated fixture cost and token observations are not billing records or
  provider invoices.
