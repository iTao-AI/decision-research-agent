# Decision Research Agent v0.1.2

## Supported Surface

v0.1.2 extends the v0.1.1 backend and Tool Client contract with optional
run-creation idempotency. The supported addition includes:

- an optional `Idempotency-Key` header on `POST /api/runs`;
- an application-database ledger for durable replay identity;
- atomic conflict and concurrent duplicate handling;
- Tool Client recovery after an ambiguous lost response; and
- a deterministic reconciliation proof with explicit boundaries.

The application database remains business authority. LangGraph checkpoints,
LangSmith traces, the Tool Client, and proof artifacts do not own idempotency
bindings or execution outcomes.

## Changes

### Run Creation Reliability

- Requests without an idempotency key remain independent and retain the
  existing response shape.
- The first keyed request creates one run and records an immutable binding.
  Reusing the same key with the same canonical request returns the original
  identity as a durable replay and schedules no second execution.
- Concurrent duplicate requests serialize through the application database so
  one request creates the run and the remaining accepted requests replay it.
- Reusing a key with different canonical request content returns
  `409 run_idempotency_conflict` without disclosing the bound run identity.
- Invalid keys return `422 run_idempotency_key_invalid`. A keyed request whose
  ledger is unavailable returns `503 run_idempotency_unavailable` and never
  falls back to an unkeyed create.
- The server stores only hashes for the key and canonical request. It does not
  persist, log, or return the raw key.

### Tool Client Lost-response Recovery

The Tool Client generates one reusable run-creation key per invocation or
accepts an explicit key. After an ambiguous timeout or connection failure, a
caller can retry the same query, profile, thread, scope, and exact key to
recover the original run identity. The client does not perform automatic
retry.

### Deterministic Reconciliation Evidence

The deterministic reconciliation proof covers lost-response replay, request
conflict, concurrent duplicate serialization, durable restart lookup, unkeyed
independence, raw-key non-persistence, and Tool Client key recovery. It is
repository-visible fixed-input contract evidence, not a provider or production
measurement.

## Compatibility And Migration

Unkeyed `POST /api/runs` callers remain compatible with v0.1.1. The canonical
result, Evidence, review, verification, publication, and profile contracts are
unchanged. The release adds no endpoint and no automatic retry policy.

The additive `007_run_create_idempotency` migration creates the
`run_create_idempotency_v1` ledger and does not rewrite existing run, Evidence,
review, verification, or publication rows. Existing schema initialization and
migration paths apply it idempotently. Operators using the explicit migration
command should retain its pre-migration backup until verification completes.

## Rollback

Rollback is an operator-controlled source and deployment action. Stop the
service, restore the v0.1.1 application revision, and restart only after health
and canonical result checks pass.

The new ledger is additive and older code does not use it. For an exact schema
rollback, restore the application database from the consistent pre-migration
backup rather than manually editing migration records or dropping tables. Keep
the application database, checkpoint database, and output storage aligned when
restoring operational state.

## Known Limits

- `crash_before_schedule_recovery: not_proven`
- `exactly_once_execution: not_claimed`
- Response loss is proven only after current-process scheduling completes.
- A handler or process interruption after commit but before scheduling is not
  recovered by this release.
- The key scope is service-wide for the current single-service
  credential/development deployment; v0.1.2 adds no tenant or actor scoping.
- The proof is deterministic local contract evidence, not a provider or production measurement,
  production SLA, external-user result, or billing record.
