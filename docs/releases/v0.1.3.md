# Decision Research Agent v0.1.3

## Supported Surface

v0.1.3 adds single-node durable run dispatch reconciliation to the v0.1.2
backend contract. The supported addition includes:

- application-owned dispatch authority persisted with each accepted run;
- recovery of committed work before Agent invocation;
- exact start fencing before the Agent, model, or tool boundary;
- startup and expired-lease reconciliation with bounded retry through three
  attempts; and
- deterministic public proof of the reviewed pre-execution recovery paths.

The application database remains business authority for dispatch, ResearchRun,
Evidence, review, verification, publication, and delivery state. Framework
checkpoints and traces do not acquire dispatch authority.

## Changes

### Durable Run Dispatch

- Run creation atomically persists a private `run_dispatches_v1` intent with
  the initial run and segment.
- FastAPI lifespan starts a single-node SQLite lease worker that reconciles
  pending work, expired leases, commit-before-schedule interruption, handler
  cancellation, and process restart before execution starts.
- Each exact claim crosses an application-owned start fence before Agent
  invocation. Stale claims cannot enter the Agent boundary.
- Scheduling is bounded to three attempts. Exhaustion converges dispatch, run,
  and initial segment to a consistent failed terminal state.
- The deterministic public proof exercises production worker, scheduler, and
  lifecycle boundaries without network or provider calls.
- `status: started` remains an acceptance acknowledgement. It does not prove
  that Agent execution has begun or completed.

## Compatibility And Migration

Existing HTTP paths and response shapes remain compatible with v0.1.2.
Canonical result, Evidence, review, verification, publication, and framework
authority boundaries are unchanged.

The additive `008_run_dispatch_reconciliation` migration creates the dispatch
ledger, uses a dedicated `.pre-run-dispatch.bak`, performs exact verification,
and restores the backup on failure. It applies no backfill and schedules no
pre-008 runs. Startup completes migration verification before starting the
dispatch worker.

## Rollback

Rollback is an operator-controlled source and data action. Stop the service
before changing application revisions. If migration 008 was applied to an
existing database, restore the complete pre-migration application database
from the dedicated backup rather than editing migration markers or dropping
tables manually. Keep application, checkpoint, and output state aligned when
restoring operational data.

## Known Limits

- exactly-once execution is not claimed.
- running execution recovery is not proven.
- provider/tool side-effect exactly-once is not claimed.
- multi-instance high availability is not proven; the supported recovery
  boundary is single-node SQLite.
- no live-provider result is observed by the deterministic proof.
- The proof is fixed-input repository evidence, not a production SLA,
  provider measurement, external-user result, or billing record.
