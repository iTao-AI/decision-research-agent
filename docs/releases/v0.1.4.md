# Decision Research Agent v0.1.4

## Supported Surface

v0.1.4 adds two bounded capabilities to the v0.1.3 supported surface:

- durable, application-owned failure causes for terminal failed runs; and
- a first-party browser Console that closes the loop from keyed run creation
  through strict status observation to the canonical result.

The application database remains business authority for dispatch, failure
causes, ResearchRun, Evidence, review, verification, publication, and delivery.
The loopback-only Console is not a publicly hosted service and does not acquire
any of that authority.

## Changes

### Durable Run Failure Causes

- Added immutable `run_failure_causes_v1` records through the additive
  `009_run_failure_cause_v1` migration.
- New terminal failures atomically persist bounded dispatch, execution, or
  finalization causes. Historical failed runs return `not_observed` rather
  than an inferred diagnosis; nonfailed runs return `null`.
- Added `failure_cause` only to `GET /api/runs/{run_id}` and retained the
  canonical result response, `409 run_failed` envelope, and frozen downstream
  fixture unchanged.
- Added a deterministic 16-case proof of the approved phase/code combinations
  and persistence boundaries.

### Console Live Authority Closure

- Kept deterministic Static Demo state isolated from Live Backend state.
- Added one immutable create intent with a stable `Idempotency-Key`; ambiguous
  responses retry only the same key and byte-equivalent body.
- Bound create acknowledgement, status, result, and structured error identity
  to the requested thread or run. Once `run_id` is known, recovery is GET-only.
- Rendered selected service-owned status fields, exact failure-cause
  combinations, observation availability, and only the canonical result.
- Retained loopback-only networking, redirect rejection, credential-free UI,
  bounded polling, unmount cancellation, and responsive bilingual controls.

## Compatibility And Migration

The status response change is additive. Existing v0.1.3 clients that ignore
unknown fields remain compatible. Result, Evidence, review, verification,
publication, delivery, profile, and idempotent create contracts are unchanged.

Migration `009_run_failure_cause_v1` adds the failure-cause ledger without
backfilling historical diagnoses. The Console adds no backend endpoint,
database authority, dependency, or deployment requirement.

## Rollback

Rollback is an operator-controlled source and data action. Stop the service
before changing application revisions. To remove migration 009 exactly,
restore the complete application database from the consistent pre-migration
backup rather than deleting tables or migration markers manually. Keep the
application database, checkpoint database, and output storage aligned.

The Console can be rolled back independently to the v0.1.3 frontend build; it
does not own durable state. Verify health, run status, and canonical result
delivery after restoring the selected revisions.

## Known Limits

- exactly-once execution, hard preemption, provider diagnosis, provider/tool
  side-effect exactly-once, and multi-instance high availability are not
  claimed.
- Historical failure causes can remain `not_observed`; the failure-cause
  contract is not a billing record or automatic root-cause analysis.
- Browser create intent is not durable browser intent across refresh or device
  loss. The service-owned run identity remains authoritative.
- Static Demo is fixture-backed presentation. Live Backend is loopback-only;
  neither proves live-provider quality, public hosting, production operation,
  production SLA, or multi-user deployment.
- The Console owns no review, verification, publication, delivery, or
  failure-cause authority.
