# Decision Research Agent v0.1.5

Release preparation date: 2026-07-18.

## Supported Surface

v0.1.5 adds secure defaults for the repository's local source and Compose
runtime paths while preserving the existing run, result, Evidence, review,
verification, dispatch, failure-cause, and downstream consumer contracts.

The supported source launcher binds to `127.0.0.1`. With an empty
`API_SECRET`, protected requests are accepted only inside the credential-free
local boundary. The direct peer and literal Host must both be loopback, and no
forwarding identity header may be present. A configured secret preserves
authenticated Tool Client and operator access through `X-API-Key`.

The supported local Compose path is authenticated. Compose requires explicit
non-empty `API_SECRET`, `MYSQL_ROOT_PASSWORD`, and `MYSQL_PASSWORD`, publishes
backend and MySQL ports only on host `127.0.0.1`, and retains the backend's
container-internal `0.0.0.0` listener inside the Compose network.

## Changes

### Secure Local Runtime Access

- The source launcher now binds to `127.0.0.1`, disables reload, and uses
  warning-level Uvicorn logging.
- Empty-secret source use now fails closed outside the direct-loopback peer,
  literal-loopback Host, and no-forwarding boundary.
- Configured credentials use constant-time comparison and continue to use the
  `X-API-Key` header.
- CORS and Origin validation remain browser boundaries rather than
  authentication.
- WebSocket credentials are header-only. Legacy `?api_key=` query credentials
  are rejected before run identity lookup or connection ownership.

### Secure Local Container Delivery

- Compose now requires explicit API and MySQL credentials and uses
  loopback-only host publication for backend and MySQL.
- MySQL health gates backend startup; the backend declares an exact
  process/service health check. That check does not claim database, provider,
  model, tool, or research readiness.
- The backend drops all Linux capabilities and enables
  `no-new-privileges`. Existing named `data` and `output` volumes remain
  writable and retain their existing format and ownership expectations.
- A deterministic proof covers 16 fixed production-path access and declared
  container-configuration cases. A separate required Docker lane covers real
  image build, health, inspection, restart persistence, no-provider behavior,
  and bounded task-owned cleanup.

## Compatibility And Migration

No API schema, database schema, migration, dependency, framework authority,
run/result/Evidence contract, or downstream fixture changes are included in
this release preparation.

- Source-loopback Console users may keep `API_SECRET` empty and continue to
  use `127.0.0.1`.
- Authenticated Tool Client and operator callers continue to send the existing
  `X-API-Key` header.
- Compose operators must set non-empty `API_SECRET`, `MYSQL_ROOT_PASSWORD`, and
  `MYSQL_PASSWORD` before `docker compose config --quiet` can succeed.
- WebSocket clients that used `?api_key=` must move the credential to the
  `X-API-Key` header. Browser clients that cannot set that header are not a
  supported authenticated WebSocket surface.
- Deliberate non-loopback use requires a configured key plus an
  operator-managed secure transport boundary; it is not a supported hosted
  deployment.

## Rollback

Stop the local service before changing application revisions. Rolling back to
v0.1.4 requires no database migration or volume-format rollback because
v0.1.5 does not rewrite schemas, named-volume data, or output artifacts.

Restoring the earlier source or Compose configuration also restores its wider
empty-secret or host-publication behavior. Treat that as a security regression,
not a recommended steady state. Keep existing application, checkpoint, and
output storage aligned when performing any independent data recovery.

## Known Limits

- This preparation document does not claim that a `v0.1.5` tag, GitHub
  Release, post-publication archive smoke, deployment, or live research has
  completed.
- This release does not provide TLS, caller identity, per-user authorization,
  RBAC, rate limiting, trusted-proxy handling, hosted production, or production deployment.
- The shared API key is a transport credential, not an identity or
  authorization model. Non-loopback plaintext transport remains unsupported.
- The backend container retains its root UID for existing named-volume
  compatibility. Capability removal and `no-new-privileges` reduce but do not
  eliminate container compromise impact.
- A compromised host or same-user local process remains inside the declared
  credential-free loopback trust boundary.
- Process health is not database, provider, model, tool, or research
  readiness.
- No live-provider research or provider-quality claim is made by the
  deterministic proof, required Docker lane, or this release preparation.
